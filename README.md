## 快速开始 (Quick Start)

### 1. 编译 C++ 核心加速库
进入 `TD-Learning+N-tuple Network` 目录，并编译不同阶段的 AI 引擎（请确保系统已安装 `g++` 并支持 OpenMP）：

```bash
cd "TD-Learning+N-tuple Network"

# 编译 base_agent
g++ -O3 -shared -fPIC -march=native -fopenmp base_agent/base_agent.cpp -o base_agent/base_agent.dll

# 编译 8k4k_agent
g++ -O3 -shared -fPIC -march=native -fopenmp 8k4k_agent/8k4k_agent.cpp -o 8k4k_agent/8k4k_agent.dll

# 编译 16k8k4k_agent
g++ -O3 -shared -fPIC -march=native -fopenmp 16k8k4k_agent/16k8k4k_agent.cpp -o 16k8k4k_agent/16k8k4k_agent.dll
```
*注：Linux/macOS 用户需将输出后缀 `.dll` 改为 `.so`，并在对应的 python 脚本中修改加载路径。*

### 2. 运行脚本
编译完成后，你可以运行以下脚本体验被 AI 统治的 2048：
- `python main_gui.py`: 启动可视化界面模式（支持自定义盘面、前进/后退、速度调节）。
- `python measure_1000_games.py`: 连续运行 1000 局并统计胜率和分数。
- `python measure_speed.py`: 测试单局运行时间与步数速度。

## 核心算法思想 (Algorithm)

### N-tuple Network
如果不考虑空间大小，使用暴力 DP 能得到最优解，但状态数 $16^{16}$ 过于庞大（假设每个格子都不超过 32768）。所以这个算法的核心是将盘面拆分成若干个 **Feature**。每个 Feature 为 16 个格子的一个子集（大小足够小，可以存入内存中），并以此训练出一个状态分数（score）。
该盘面的总 score 即为所有 Feature 的 score 之和。

### TD-Learning (时序差分)
模型使用时序差分更新，AI 每走一步都会更新一次参数。
假设该步动作的直接得分为 $inc$，动作前后的盘面 score 分别为 $s_1, s_2$。系统会给动作前盘面的每个 Feature 的 score 加上 $\alpha\cdot (s_2+inc-s_1)$。这里 $\alpha$ 为一个极小的学习率常数，代码里取值 $\alpha=0.001$。

### Expectimax Search (期望最大值搜索)
给定搜索深度 `depth`，计算目前盘面往四个方向走一步之后，再经过 `depth` 回合内（假设每步都走最优）的期望分数（递归到 `depth=0` 时直接调用盘面模型估测的 score），最后取期望分数最大的 Action 作为当前动作。
显然，`depth` 越大模型越强，但由于每次落子都会产生随机块，搜索树大小呈指数级爆炸，速度会下降很多。

## Multi-Stage 多阶段训练策略
本项目使用了**多阶段训练方法**，将一局 2048 游戏拆分成了三个阶段（定义盘面上 $\ge 4096$ 的数为“大数”，设其数量为 $A$）：
- **$A \le 1$**：使用 `base_agent` 模型。
- **$A = 2$**：使用 `8k4k_agent` 模型。
- **$A \ge 3$**：使用 `16k8k4k_agent` 模型。
*(注：在训练 16k8k4k 模型时只考虑了 $A=3$ 的情形，因此当盘面同时出现 32768, 16384, 8192, 4096 时 AI 会陷入混乱。这也是为什么目前最高分大约止步于 820,000 左右)*。

## 模型与特征设计 (Model Details)
模型总大小约 **520MB**。具体设计细节如下：

### 1. Tuple 设计
假设盘面 16 个格子编号依次为 0-15：
- **`base` 模型**使用了 6 个 6-tuple：`{0,1,2,3,4,5}`, `{4,5,6,7,8,9}`, `{8,9,10,11,12,13}`, `{0,1,2,4,5,6}`, `{4,5,6,8,9,10}`, `{0,1,2,3,4,8}`。
- **`8k4k` 模型和 `16k8k4k` 模型**多使用了一个 6-tuple，共计 7 个：增加了 `{4,5,6,7,8,12}`（L形阵形）。
- 每个 Feature 取自所有 tuple 经过盘面旋转、翻转后的所有组合。由于旋转和翻转不影响策略逻辑，相同派生 tuple 在内存中共享同一块地址，得分也完全一致。

**设计合理性：**
目前网上的 AI 使用的阵形大多数为 `628` 或 `63L`。作者查阅胜率数据后发现，在 32k 残局 (final 512) 中，蛇形层叠阵形的胜率并不逊色，因此围绕**层叠阵形**进行 tuple 设计：

- 前三个 tuple 是最为标准的层叠形状。
- 后两个矩形 tuple 主要起到**防滑作用**。（例如某行是 `2048,1024,32,16`，防滑设计能引导其合并为 `2048,1024,64,16` 而非错误的 `...32,32`）
- L 形 tuple 是为了学习**滑角后的救场机制**。（例如角落的 `64` 滑出变成 `2048,1024,256,32,0,0,64,2` ，此时最优策略就是从 `2` 的下面继续合成） 在第三个阶段中，当第一行被大数填满时，盘面仅剩 3 行进行游戏，两个 L 形的存在能显著提升极端情况的存活率。

### 2. 模型目标、初始状态与终止截断
所有模型均存储在各自 Feature 对应的 score 数组中，初始化参数全为 0。为了防止 AI "死在莫名奇妙的地方导致负向更新"（例如专门训练两大数残局的 8k4k 模型却死在了 16k 加上一堆小数里），我们在训练中引入了**截断机制**，实测提升显著。

| 模型 | 训练目标 | 初始盘面分布 | 终止条件 (截断) |
| --- | --- | --- | --- |
| **Base** | 从 0 到 4k，以及单大数环境下，再合出 4k | 标准 2048 初始盘面 (两个 2 或 4) | 合出 8k，或正常死亡 |
| **8k4k** | 两个不同大数环境下，再合出 4k | 90% 概率角落排布(16k, 4k，其他 $\le 16$)；10% 随机排布(训练乱阵归位能力) | 合出 16k+8k，或正常死亡 |
| **16k8k4k**| 三个不同大数环境下，再合出 4k | 90% 概率角落排布(32k, 16k, 4k，其他 $\le 16$)；10% 随机排布(训练乱阵归位能力) | 合出 32k+16k+8k，或正常死亡|

### 3. 训练细节秘籍
1. **训练参数**：采用阶段性学习率（$\alpha$）衰减策略。
   | 模型 | 总训练局数 | 阶段 1 ($\alpha=0.001$) | 阶段 2 ($\alpha=0.0002$) | 阶段 3 ($\alpha=0.00004$) | 阶段 4 ($\alpha=0.00001$) |
   | --- | --- | --- | --- | --- | --- |
   | **Base** | 50M 局 | 前 40M 局 | 后 8M 局 | 后 1.6M 局 | 最后 0.4M 局 |
   | **8k4k** | 75M 局 | 前 60M 局 | 后 12M 局 | 后 2.4M 局 | 最后 0.6M 局 |
   | **16k8k4k** | 100M 局 | 前 80M 局 | 后 16M 局 | 后 3.2M 局 | 最后 0.8M 局 |
2. **极速训练**：训练速度为 `1000000 steps/s` 以上，跑完 10000 局仅需几十秒。
3. **初始池刷新**：每次生成 1000 个初始盘面供随机抽取，每打 1 万局剔除并换新 2 个盘面（有效防止过拟合）。
4. **特殊判断**：在大数合并阶段（单步分数大于 4000 分）尽量保证最大数在角落，防止可能的大数归位损失成功率。
5. **盘面降维与最终奖励**：在触发死亡截断前的最后一步更新至关重要。如果直接给成功事件一个超大 Constant Reward 实测效果不佳。更好的做法是：**将最后截断状态“降维”后的score加上该步分数作为其估值**（把 8192 降级视作 4096）。因为终止状态在降维后就等价于该模型训练时的初始盘面，所以能非常自然、合理地得到分数。
6. **模型极致压缩 (进制缩维)**：
   - **Base (~110MB)**：盘面上 $\ge 4096$ 的数没有本质区别，最高统计到 4096。将 16 进制缩减至 **13 进制**，参数量从 $6\times 16^6$ float暴降至 $6\times 13^6$ float。
   - **8k4k (~200MB)**：盘面上 $\ge 8192$ 的数无区别，采用 **14 进制**。*(注：故意保留了 4096 与 8192 的区分度，防止 AI 利用对称性将 4096 和 8192 强行挤在中间而非角落)*。
   - **16k8k4k (~200MB)**：同理采用 **14 进制**。$16384$ 与 $8192$ 则因其位置绝对固定在角落(0和1)，在翻转/旋转下不会冲突，因此无需拆分。

## 程序加速与性能优化 (Optimization)
本项目主要进行了四维度的底层优化：

1. **Bitboard & 查表法**
   不考虑极端的 65536 块，每个格子有 16 种状态，正好占 `4-bit`。
   整个 4x4 盘面直接压缩成一个 64 位的 `unsigned long long`。
   预先计算出单行（共 $16^4 = 65536$ 种组合）的左右移动位移变换和得分增量，用空间换时间。这使得对盘面进行一轮拨动只需查表 **4** 次。
2. **哈希去重 (Transposition Table)**
   在 Expectimax 展开搜索树中，"先上后左" 和 "先左后上" 往往导致相同的盘面结果。利用 Hash 去重有效剪枝。
3. **四路并行展开 (Multi-Threading)**
   针对于对弈树的根节点（即真实操控的四个方向步），利用 OpenMP 开启 4 个线程并行递归搜索。
4. **无锁哈希表 (Lock-Free Transposition Table)**：与传统的加锁 (Mutex) 不同，程序使用了高阶的 C++ `std::atomic` 原子操作和 `std::memory_order_relaxed` 内存序来进行并发读写。在保证线程安全不冲突的同时，完全消除了锁竞争开销，完美榨干现代 CPU 的多核性能。

加上以上四个优化后，在四层搜索下（depth=3），运行速度从 **100** steps/s 提升到了 **750** steps/s。
